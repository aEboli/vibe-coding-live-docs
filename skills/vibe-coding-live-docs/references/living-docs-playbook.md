# Vibe Coding Playbook

## 目的

这份手册用于把 Vibe Coding 开发流程从“开工前一次性起草”改成“跟着项目演进的活文档体系”。

- 默认触发场景：新项目、接手陌生项目、较大功能、核心流程、需要跨页面或跨模块协同的改动。
- 精简触发场景：小修复、小改文案、小范围 UI 微调、明确边界内的单点修改。
- 总原则：默认遵守，但按任务规模缩放；做不了了不强行做，优化不了了不强行优化。

## 文档体系总览

### 主文档

- `project-docs/PRD.md`
  - 负责“做什么”和“怎么验收”。
  - 保留产品目标、目标用户、关键流程、验收标准和页面摘要。
- `project-docs/ARCHITECTURE.md`
  - 负责“怎么实现”和“为什么这样落地”。
  - 保留当前架构结论、模块职责、关键数据流、服务端逻辑和真实落地结构。
- `project-docs/PROJECT_STATE.md`
  - 负责“当前推进到哪里、哪些文档还没补齐、下一次该补什么”。
  - 它是文档同步总控台，不只是开发进度表。

### 附录文档

- `project-docs/PAGE_FRAME.md`
  - UI 或多页面项目启用。
  - 集中维护页面清单、导航结构、页面状态、跳转关系、关键参考。
- `project-docs/TECH_DECISIONS.md`
  - 存在非平凡技术选择时启用。
  - 集中维护技术栈定稿、关键库选择、放弃方案、约束原因和确认时间。

### 支撑文档

- `project-docs/DEVELOPMENT_RULES.md`
  - 记录语言/框架约束、模块拆分约束、修改边界、安全底线和调试规则。
- `project-docs/QUALITY_GATE.md`
  - 记录版本管理、验证要求、代码审查、发布前检查和文档收敛检查。
- `project-docs/references/`
  - 存放可信参考资料、实现示例、设计参考和外部来源。

## 三阶段活文档

所有主文档与附录文档都遵循同一生命周期：

### `Draft`

- 含义：项目启动时的骨架版本。
- 允许内容：
  - 候选技术栈
  - 待确认模块边界
  - 页面草图或导航草案
  - 待确认风险
- 不要求：
  - 一次写死最终技术细节
  - 一次写完全部页面和全部模块

### `Confirmed`

- 含义：关键细节已经稳定，可以作为当前执行依据。
- 进入条件：
  - 至少有首个可运行版本，或关键事件已定稿。
  - 相关文档中的“仍待确认项”已明显减少。
- 要求：
  - 关键信息从“候选”转为“已确认”
  - 当前团队或 AI 后续执行时可直接参考

### `As-Built`

- 含义：按实际落地结果收敛后的版本。
- 进入条件：
  - 准备发布，或阶段性开发已稳定收尾。
- 要求：
  - 记录最终采用方案
  - 记录与原计划差异
  - 记录删减项与剩余风险

## 双触发更新机制

### 里程碑触发

以下节点必须回填文档：

1. 项目启动
2. 首个可运行版本
3. 核心流程稳定
4. 准备发布

### 事件触发

以下事件发生时，不等到下一个里程碑，也要即时更新对应文档：

- 技术栈定稿
- 页面框架定稿
- 核心模块边界变化
- 鉴权/数据流变化
- 第三方依赖或服务选择落定

## 全流程如何改成“先建骨架，后续回填”

### 阶段 1：定图纸

#### 步骤 1：先倒需求，再写代码

- 何时触发：任何新项目或大功能立项时。
- 最少要问什么：
  - 目标用户是谁。
  - 核心功能是什么。
  - 最重要的使用场景是什么。
- 最少要产出什么：
  - 一段需求摘要。
  - 一个可验证的目标描述。
- 生命周期要求：
  - 写入 `project-docs/PRD.md` 的 `Draft` 骨架。
  - 明确哪些问题尚未确认，不要求一次定完。

#### 步骤 2：整理 PRD 并定初版验收标准

- 何时触发：需求摘要能说清后立刻触发。
- 最少要问什么：
  - 功能列表包含哪些核心能力。
  - 用户流程从哪开始、到哪结束。
  - 每个功能怎样算完成。
- 最少要产出什么：
  - `project-docs/PRD.md` 的 `Draft` 版本。
  - 初版验收标准和待确认项。
- 推荐子技能：
  - 当输入来源是 feature idea、meeting notes、stakeholder ask、rough brief、research summary 时，优先用 `$prd-creater` 产出结构化首版 PRD。
  - 当已经有成熟 PRD、小任务只需要最小状态摘要、或用户不想要 Notion 风格输出时，不强制调用 `$prd-creater`。
- 生命周期要求：
  - 到“核心流程稳定”时，把初版验收标准收敛到 `Confirmed`。
  - 到“准备发布”时，补实际落地偏差到 `As-Built`。
- 职责边界：
  - `$prd-creater` 负责把零散输入整理成结构化 PRD 草稿，并借 Notion 官方/模板体系补骨架。
  - `vibe-coding-live-docs` 负责把草稿映射到 `project-docs/PRD.md`，补生命周期字段、页面摘要，并继续和架构/状态文档联动。

#### 步骤 3：先定视觉方向，再逐步收敛页面框架

- 何时触发：涉及页面、交互、展示层时。
- 最少要问什么：
  - 视觉风格偏向什么方向。
  - 页面框架是否已有约束。
  - 有没有参考网站或现成设计系统。
- 最少要产出什么：
  - `project-docs/PRD.md` 中的页面/导航摘要。
  - UI 项目的 `project-docs/PAGE_FRAME.md` 骨架。
- 生命周期要求：
  - `Draft` 只要求页面草图、导航草案、关键页面列表。
  - 首个可运行版本后回填真实页面清单、导航和状态。
  - 核心流程稳定后收敛为 `Confirmed`。

### 阶段 2：打地基

#### 步骤 4：明确项目边界和非功能需求

- 何时触发：进入技术选型前。
- 最少要问什么：
  - 自用还是公开。
  - 是否涉及用户数据或权限。
  - 性能、成本、安全、可用性是否有限制。
- 最少要产出什么：
  - 边界说明。
  - 非功能需求列表。
- 生命周期要求：
  - 先写入 `project-docs/ARCHITECTURE.md` 的 `Draft` 区域。
  - 只要鉴权、数据流或性能约束发生变化，必须事件触发回填。

#### 步骤 5：锁定可验证技术栈，但允许先保留候选方案

- 何时触发：确定边界后。
- 最少要问什么：
  - 当前团队或项目已有技术约束是什么。
  - 哪套方案最容易验证、资料最全、工具链最成熟。
- 最少要产出什么：
  - `project-docs/ARCHITECTURE.md` 中的“候选 vs 已确认技术栈”。
  - 非简单项目的 `project-docs/TECH_DECISIONS.md` 骨架。
- 生命周期要求：
  - `Draft` 允许保留候选方案。
  - 技术栈一旦定稿，立即写进 `project-docs/TECH_DECISIONS.md`，并同步 `project-docs/ARCHITECTURE.md` 当前结论。

#### 步骤 6：让架构从草案走向真实落地

- 何时触发：技术栈锁定后，写代码前和写代码过程中都可能触发。
- 最少要问什么：
  - 目录怎么分层。
  - 核心模块有哪些。
  - 数据模型和服务端职责怎么划。
- 最少要产出什么：
  - `project-docs/ARCHITECTURE.md` 的模块边界草案。
  - `project-docs/PROJECT_STATE.md` 里的文档待补项。
- 生命周期要求：
  - 首个可运行版本后回填初版真实目录和模块。
  - 核心模块边界变化时，立即更新 `project-docs/ARCHITECTURE.md` 和 `project-docs/PROJECT_STATE.md`。
  - 发布前补 `As-Built` 结构和偏差原因。

### 阶段 3：立规矩

#### 步骤 7：固化文档骨架，并让 `PROJECT_STATE` 管理文档债务

- 何时触发：项目启动时立即触发，最晚不超过第一个切片完成前。
- 最少要产出什么：
  - `project-docs/PRD.md`
  - `project-docs/ARCHITECTURE.md`
  - `project-docs/PROJECT_STATE.md`
- 生命周期要求：
  - `project-docs/PROJECT_STATE.md` 必须增加：
    - 文档待补项
    - 最近确认事项
    - 下一次回填节点
    - 回填责任文档
    - 已冻结决策

#### 步骤 8：定开发规范 + 建参考资料夹

- 何时触发：准备正式开发前。
- 最少要产出什么：
  - `project-docs/DEVELOPMENT_RULES.md`
  - `project-docs/references/`
- 生命周期要求：
  - 如果项目中新增特殊约束或可信参考，要跟着项目迭代同步补充，而不是只在开头写一次。

#### 步骤 9：写代码前先搭 Git 和质量闸门，并把文档收敛纳入闸门

- 何时触发：第一次提交前。
- 最少要产出什么：
  - `project-docs/QUALITY_GATE.md`
- 生命周期要求：
  - 把文档状态检查纳入质量闸门：
    - 项目启动后是否已创建 `Draft` 骨架
    - 首个可运行版本后是否已回填页面/技术/模块
    - 发布前是否已收敛成 `As-Built`

## 每份文档怎么写才更完整

### `project-docs/PRD.md`

- `Draft` 至少包含：
  - 目标用户
  - 核心目标
  - Goals and KPIs / 目标与指标
  - Constraints and assumptions / 约束与假设
  - Dependencies / 依赖
  - 核心流程
  - 初版完成标准
  - 视觉方向
  - 页面/导航摘要
- `Confirmed` 必须补齐：
  - 页面清单
  - 导航结构
  - 关键页面状态
  - 异常流和失败提示
  - 最终验收口径
- `As-Built` 必须补齐：
  - 实际落地偏差
  - 删减项
  - 保留风险

### `project-docs/ARCHITECTURE.md`

- `Draft` 至少包含：
  - 候选技术栈
  - 候选模块边界
  - 待确认数据模型
  - 初版服务端逻辑
- `Confirmed` 必须补齐：
  - 已确认技术栈
  - 核心模块职责
  - 模块关系
  - 关键数据流
  - 服务端职责
  - 鉴权授权方案
- `As-Built` 必须补齐：
  - 真实目录结构
  - 真实模块和依赖
  - 偏离原草案的原因

### `project-docs/PROJECT_STATE.md`

- 必须长期维护：
  - 当前目标
  - 当前切片
  - 范围边界
  - 文档待补项
  - 最近确认事项
  - 下一次回填节点
  - 回填责任文档
  - 已冻结决策
  - 验证结果
  - 下一步

### `PAGE_FRAME.md`

- 适用场景：
  - 多页面项目
  - 有明确导航或复杂页面状态的项目
- 必须集中维护：
  - 页面清单
  - 导航结构
  - 页面间跳转
  - 关键页面状态
  - 参考截图或参考链接

### `TECH_DECISIONS.md`

- 适用场景：
  - 技术栈存在取舍
  - 依赖或服务选型不是显而易见
- 必须集中维护：
  - 当前采用方案
  - 放弃方案
  - 约束原因
  - 决策日期
  - 对应影响范围

## 更新清单

### 项目启动

- 创建 `project-docs/PRD.md`、`project-docs/ARCHITECTURE.md`、`project-docs/PROJECT_STATE.md` 的 `Draft` 骨架。
- UI 项目创建 `project-docs/PAGE_FRAME.md`。
- 技术不简单的项目创建 `project-docs/TECH_DECISIONS.md`。

### 首个可运行版本

- 回填真实页面框架。
- 回填首版技术栈。
- 回填初版核心模块。
- 在 `project-docs/PROJECT_STATE.md` 中清掉一批文档待补项。

### 核心流程稳定

- 收敛验收标准。
- 收敛核心模块职责和数据流。
- 收敛关键技术决策。
- 将核心文档状态推进到 `Confirmed`。

### 准备发布

- 将主文档收敛为 `As-Built`。
- 补齐实际偏差和剩余风险。
- 在 `project-docs/QUALITY_GATE.md` 里完成文档收敛检查。

## 开发中 5 个核心关键点

### 1. 小步迭代 + 端到端切片验证

- 任何需求都优先拆成可验证切片。
- 每完成一个切片就做一次端到端验证，再进入下一个切片。
- 验证结果要同步进 `project-docs/PROJECT_STATE.md`，而不是只留在聊天记录里。

### 2. 人类主动拆分模块，禁止“一坨代码”

- 每次开工前先明确边界和文件职责。
- 如果一个文件已经承担过多职责，应在当前任务范围内做最小必要拆分。
- 不为了省事把新增逻辑全部堆到单文件里。

### 3. 限制 AI 擅自做主，用状态摘要管理上下文

- 未点名的文件和功能默认不改。
- 不擅自重构、迁移、顺手优化无关逻辑。
- 每轮开发前后都用状态摘要和文档待补项确认目标、范围、进度、风险和下一步。

### 4. 严守安全底线，代码不写死敏感信息

- API Key、密钥、令牌、数据库密码等不允许硬编码到前端或仓库。
- 默认使用环境变量、服务端代理和鉴权授权校验。
- 只要涉及用户数据，就必须同步回填数据流和访问控制文档。

### 5. 科学应对报错，两次新增证据无效则停手

- 先最小化复现，再打印关键变量、关键分支和上下文。
- 修改必须基于新增证据，而不是盲猜。
- 如果连续两轮新增证据后仍没有实质进展，停止蛮干，建议回到最近稳定点重新判断方向。

## 精简版流程

以下情况默认走精简版：

- bugfix
- 小范围文案或样式调整
- 已有架构下的单点功能补丁
- 明确只改少量文件的任务

精简版最少执行 5 步：

1. 写清目标和完成标准。
2. 写清修改范围和不改范围。
3. 检查是否涉及敏感信息、数据流、权限或副作用。
4. 拆成最小切片并逐个验证。
5. 更新 `project-docs/PROJECT_STATE.md` 或最小状态摘要。

如果精简任务实际触发以下变化，必须升级更新对应文档：

- 页面框架变化 -> 更新 `PAGE_FRAME.md` 和 `project-docs/PRD.md` 摘要
- 技术栈变化 -> 更新 `TECH_DECISIONS.md` 和 `project-docs/ARCHITECTURE.md`
- 模块边界变化 -> 更新 `project-docs/ARCHITECTURE.md` 和 `project-docs/PROJECT_STATE.md`

## 止损规则

满足以下任一条件时，不继续硬做：

- 需求、边界或验收标准长期不清。
- 连续两轮新增证据后仍无法推进。
- 当前改动已经明显超出原定范围。
- 为了“先跑起来”开始引入硬编码敏感信息、跳过鉴权或绕过关键验证。
- 优化收益不明确，但复杂度和风险持续上升。

遇到止损条件时，默认动作：

1. 停止继续扩散改动。
2. 记录当前证据、尝试过的路径和阻塞点。
3. 回到最近稳定点，重新确认需求、范围或方案。
